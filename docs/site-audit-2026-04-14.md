## Критические и высокие риски

### 🔴 1. Публичный диагностический endpoint платежей без защиты

`/api/payment/check` выполняет:
- проверку магазина (`/v3/me`),
- создание тестового платежа (`/v3/payments`),
- и возвращает диагностику в ответ.

Риски:
- endpoint доступен любому внешнему клиенту;
- можно дергать массово и создавать нагрузку/шум в платежной системе;
- в ответе раскрываются детали состояния платежной конфигурации.

### 🔴 2. Webhook платежей не валидирует источник уведомления

В `api/payment/webhook` payload принимается как есть и обрабатывается только по `event === payment.succeeded`; проверки подписи/источника webhook нет.

Риск: теоретически можно отправить поддельный POST-запрос и триггерить бизнес-логику/уведомления.

### 🟠 3. Секретный доступ к платному отчету через слабый fallback-ключ

Доступ к `/benchmark/report` защищен query-ключом, но есть дефолт: `2026q1`, если env-переменная не задана.

Риск: при невыставленном `REPORT_ACCESS_KEY` доступ становится предсказуемым.

### 🟠 4. Жестко задан fallback topic для ntfy

`NTFY_TOPIC` имеет дефолтное значение в коде. При невыставленном env-пространстве может утекать операционная информация в общий/предсказуемый канал.

## Средние риски и качество

### 🟡 5. Большой объем линт-ошибок в прод-коде

50 ошибок lint — это реальный риск регрессий, проблем поддержки и нестабильного DX.

### 🟡 6. Нестабильность build в ограниченной сети

Использование `next/font/google` без fallback на локальные шрифты может ломать сборку в CI/изолированных окружениях.

## Рекомендуемый порядок исправления

1. Закрыть `/api/payment/check` за auth (ключ в header, basic auth, allowlist IP или отключение в production).
2. Добавить валидацию подлинности webhook (подпись/верификация через YooKassa API).
3. Убрать слабые fallback для секретов (`REPORT_ACCESS_KEY`, `NTFY_TOPIC`) и падать с 5xx, если они не заданы.
4. Погасить линт-критичные ошибки (в первую очередь `no-html-link-for-pages`, `no-explicit-any`, `hooks/purity`).
5. Рассмотреть `next/font/local` или гарантировать сетевую доступность Google Fonts в CI.

## Вердикт

Сайт функционально близок к рабочему, но **в текущем состоянии не проходит базовый quality gate** из-за:
- большого количества lint-ошибок,
- падения production build в ограниченной сети,
- и нескольких рисков безопасности в платежной части.

Для production-hardening рекомендуется закрыть security-риски и пройти «чистый» lint/build перед следующей выкладкой.

## Что проверено дополнительно

2. Карта сайта (`sitemap.ts`) и `robots.txt`.

4. Покрытие страниц метаданными (скрипт по `app/**/page.tsx`).

## SEO: сильные стороны

### ✅ 1. Хорошее покрытие метаданными на большинстве страниц

- В проекте 174 `page.tsx`.
- У подавляющего большинства страниц есть `metadata`/`generateMetadata`.
- Для статей массово используются `canonical`, `openGraph`, `twitter`.

Это сильная база для индексации и CTR в выдаче.

### ✅ 2. Наличие структурированных данных

- Используются JSON-LD схемы `Article`, `FAQPage`, `HowTo`, `WebApplication`.
- Для контентного SEO это плюс (rich snippets, better entity understanding).

### ✅ 3. Широкое long-tail покрытие

- В sitemap явно генерируются кластеры страниц по профессии/городу/уровню.
- Это правильно для захвата низкочастотного спроса.

## SEO: проблемы и риски

### 🔴 1. Глобальный `FAQPage` в `layout.tsx` (на всех URL)

Сейчас FAQ schema находится в корневом layout, значит попадает **на каждую страницу сайта**, включая сервисные и узкотематические URL.

Почему это риск:
- поисковику подается нерелевантная структурированная разметка для части страниц;
- возрастает вероятность снижения доверия к schema (spam/irrelevant structured data).

Рекомендация:
- убрать `FAQPage` из глобального layout;
- оставлять FAQ schema только на страницах, где этот FAQ реально присутствует в контенте.

### 🔴 2. Утечка платного URL в публичной странице оплаты

На `/payment/success` зашит прямой URL отчета с query-ключом: `.../benchmark/report?key=2026q1`.

Почему это риск:
- ключ оказывается в клиентском коде/HTML;
- ссылка может утечь через индексацию, рефереры, скриншоты, шеры.

Рекомендация:
- не хранить приватный ключ в клиентском бандле;
- выдавать доступ через одноразовый/подписанный токен с TTL или через серверную сессию;
- закрыть `/payment/success` метаданными `robots: noindex`.

### 🟠 3. Неактуальные/жестко зашитые коммерческие формулировки по дате

В post-payment контенте встречаются фиксированные даты и кварталы (`Q2 2026`, `Q3 2026`), а событие цены привязано к 7 апреля 2026.

Почему это риск:
- контент стареет, снижается trust и конверсия;
- поисковик видит устаревающие коммерческие блоки на evergreen URL.

Рекомендация:
- вынести даты/кварталы в конфиг;
- добавить процесс контентного обновления (ежеквартально, с checklist).

### 🟠 4. Слабая надежность build-фактора для SEO (доступность релиза)

Сборка зависит от внешней загрузки шрифта Google.

Почему это риск:
- любой сетевой сбой в CI/CD блокирует релиз;
- задержка релизов = задержка индексации новых страниц.

Рекомендация:
- перейти на `next/font/local` для ключевого шрифта, либо обеспечить стабильный доступ в CI.

### 🟡 5. `sitemap` выставляет `lastModified: new Date()` для всех URL

При каждом билде все страницы выглядят «обновлёнными сейчас».

Почему это риск:
- сигнал свежести может стать шумным и недостоверным;
- crawl budget расходуется менее эффективно.

Рекомендация:
- подставлять реальные даты обновления (из контента/файла/данных), хотя бы по крупным разделам.

## Смысловой контент: оценка

### Что хорошо

- Материалы структурированы под интент пользователя: «можно ли», «сколько», «как рассчитать».
- В статьях есть формулы, таблицы, примеры и FAQ-блоки.
- Сильная связка с продуктом (встроенные CTA к калькуляторам/сравнениям).

### Что проседает

1. **Недостаток прозрачности источников цифр.**  
   В статьях много конкретных процентов/диапазонов ставок, но без явной библиографии/ссылок на первичные данные.

2. **Риск шаблонной повторяемости.**  
   Большой массив типовых SEO-страниц может восприниматься как semi-duplicate без уникальных данных/кейсов на каждой странице.

3. **Слабая E-E-A-T-подача для YMYL-подобных тем (налоги/право).**  
   Есть юридические тезисы, но нет явного блока «кто автор/эксперт», «когда проверено», «дисклеймер».

## Приоритетный план именно для SEO+контента

1. Убрать глобальный FAQ schema из `layout` (оставить page-level).
2. Исправить post-payment flow: noindex + убрать ключ из клиента + серверная выдача отчета.
3. Перевести `lastModified` в sitemap на реальные даты.
4. Для налоговых/правовых страниц добавить блоки:
   - «Проверено редакцией / дата проверки»,
   - «Нормативные ссылки (ФНС, закон, письма)»,
   - «Дисклеймер: не индивидуальная консультация».
5. Ввести контент-матрицу обновлений (по кварталам) для ставок, налогов и офферов.
